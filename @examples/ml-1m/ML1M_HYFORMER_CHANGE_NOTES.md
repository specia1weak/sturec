# ML1M HyFormer Change Notes

## Effective changes

This document records two model changes on top of `@examples/ml-1m/ml1m-hyformer.py`
that showed clear gains in practice.

### 1. Query seed no longer comes from sequence mean pooling

Previous behavior:

- `QueryGenerator` pooled `seq_tokens` with masked mean pooling.
- The pooled result was concatenated with flattened NS tokens.
- Query seed was therefore influenced by both content tokens and time-enhanced
  sequence tokens before any cross-attention happened.

Current behavior:

- `QueryGenerator` only consumes flattened NS tokens.
- NS tokens come from:
  - `user_group_token`
  - `item_group_token`
- Query seed becomes a current-context retrieval anchor, instead of a summary
  vector averaged from the whole sequence.

Reasoning:

- Mean pooling weakens recency and order signals.
- In this implementation, sequence tokens already include relative time
  encoding, so pooling them also averages time information into the query seed.
- Letting the query start from current context and then read sequence via
  cross-attention is structurally cleaner.

### 2. Use two query tokens and restore full RankMixer

Previous behavior:

- `num_queries = 1`
- `num_ns_tokens = 2`
- total token count into mixer was `3`
- with `d_model = 16`, full token mixing was illegal because `16 % 3 != 0`
- model had to fall back to `ffn_only`

Current behavior:

- `num_queries = 2`
- `num_ns_tokens = 2`
- total token count into mixer is now `4`
- with `d_model = 16`, `16 % 4 == 0`
- `RankMixer(full)` can be used again

Reasoning:

- RankMixer becomes much more meaningful once token count is compatible with
  channel rewiring.
- Two query tokens provide more room for interaction with two NS tokens.
- This change preserves `d_model = 16` and `num_heads = 4`, so attention stays
  on a stable configuration while mixer regains its original mechanism.

## Auxiliary CTR route from query seed

The current auxiliary CTR supervision does **not** make each query predict CTR
independently.

Current implementation:

- Two initial query tokens are generated by `QueryGenerator`.
- They are flattened into one vector of shape `B x (num_queries * d_model)`.
- A single `query_seed_head` maps that joint seed representation to one CTR
  logit.
- Training adds an auxiliary BCE loss on this single logit.

Why this was chosen:

- It keeps the auxiliary task aligned with the final CTR target.
- It avoids forcing artificial symmetry where each query must solve CTR alone.
- It treats the two query tokens as a coordinated seed representation.

Possible next variant:

- Add per-query auxiliary heads and encourage diversity between the two query
  tokens.
- This was not enabled in the current version because the first goal was to
  verify whether the joint seed route is useful at all.

## Current interpretation

The current model should be read as:

1. Build two NS tokens from current user/item grouped context.
2. Generate two query seed tokens only from the current context.
3. Use those query tokens to read the sequence through HyFormer blocks.
4. Let RankMixer full mixing operate over:
   - 2 query tokens
   - 2 NS tokens
5. Supervise:
   - the final CTR route
   - the query-seed auxiliary CTR route

## Training observation on ML-1M

One practical observation from the ML-1M setting:

- sequence-side improvements may need more epochs before their gains become
  visible;
- early or mid-stage validation can easily underestimate sequence modeling
  changes.

This is likely related to model-to-data scale mismatch:

- the official-style HyFormer setup was designed with much larger sequence
  configurations and a heavier backbone;
- on a dataset around MovieLens-1M scale, that default style can be too large
  relative to the amount of data;
- as a result, useful sequence changes may look weak at first because training
  is slower, optimization is noisier, and the model can spend a long time
  fitting easier non-sequence signals first.

Current takeaway:

- when evaluating sequence modeling changes on ML-1M, avoid concluding too
  early from short training runs;
- sequence-related gains may only become clear after more training epochs than
  simple non-sequence tweaks;
- official large-sequence defaults should not be assumed to transfer cleanly to
  1M-scale data without downsizing.

## Additional hypothesis on feature initialization

This is currently a working hypothesis rather than a confirmed conclusion.

Hypothesis:

- the official-style setup should probably avoid unclear dense features unless
  they are truly meaningful vector-dense inputs;
- apart from a few genuine vector-dense features, other weakly-defined dense
  features may be better initialized at a much smaller common scale such as
  `1e-3`, so that feature magnitudes start from a more uniform range.

Reasoning behind the hypothesis:

- when heterogeneous feature groups start with very different effective scales,
  optimization can become noisy;
- some dense inputs may dominate early training even if their semantics are not
  stronger than sparse/id-style signals;
- on smaller datasets such as ML-1M, that scale mismatch may create additional
  instability and make the learned feature balance harder to interpret.

Current status:

- this point is recorded as an empirical suspicion that still needs controlled
  ablation;
- it should not yet be treated as a settled conclusion.

## Strong observation on sigmoid feature scaling

Another strong empirical observation from the current ML-1M experiments:

- applying a static sigmoid scaling gate directly on non-sequence feature
  embeddings is an unusually robust recommendation-system trick;
- in practice it tends to smooth optimization reliably, and it is much more
  broadly useful than putting the same sigmoid gate inside every projector.

Observed pattern in this project:

- sigmoid gates inside all projector/down-projection modules mainly improved
  training smoothness, but did not raise the performance ceiling;
- moving the sigmoid gate to the non-sequence feature embeddings themselves was
  the more convincing usage pattern.

Current interpretation:

- the most reusable place for this trick is at the feature-input level;
- sequence projection and token projector internals do not necessarily benefit
  from carrying the same gate;
- if a future experiment wants to reuse this idea, the default first attempt
  should be: gate non-sequence embeddings once, then reuse the scaled features
  everywhere downstream.

## Independent appended-query auxiliary branch

An additional auxiliary sequence branch is now kept separate from the main
HyFormer route.

Current implementation:

- take the two `query_seed_tokens` produced from NS context only;
- append both query tokens to the end of the time-enhanced sequence;
- assign both appended query tokens the current interaction time, so their
  relative-time embedding corresponds to `delta = 0`;
- run one plain `SequenceEvolutionBlock` on this extended sequence;
- take the last two positions together and concatenate them into one vector for
  auxiliary CTR prediction.

Why this version is preferred over the previous draft:

- the previous draft appended two query tokens but only supervised the final
  one, which created asymmetric learning pressure;
- concatenating the last two outputs preserves the distinct roles of the two
  query tokens instead of averaging them away;
- keeping this branch independent avoids mixing its effect into the main
  HyFormer path too early, which makes ablation cleaner.
