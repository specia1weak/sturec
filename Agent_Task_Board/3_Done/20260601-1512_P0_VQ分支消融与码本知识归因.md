# VQ 分支消融与码本知识归因

**发布时间**: 2026-06-01 15:12
**发起者**: Agent 1

## 1. 背景与现状分析
`vq_share_gate_v1` 已经在 KuaiRand-1K 上跑出 `0.78621 ~ 0.78662` 的有效提升，但收益来源还不清楚：

- 是 `shared` 主干本身更强，还是 `specific` 分支在头部场景放大了收益。
- VQ codebook 存的是“真正跨场景共享的知识”，还是被大场景样本主导后形成的伪共享。
- 为什么 epoch1 -> epoch2 里大场景上涨，而小场景尤其 `domain4` 更容易掉点。

本轮目标不是继续堆新结构，而是先把现有收益的来源拆清楚，再决定下一步该改哪一侧。

## 2. 明确的执行目标

- [ ] 校验当前 KuaiRand 训练入口的真实评估 split，确认结论对应的数据切分。
- [ ] 统计 VQ codebook 的跨场景使用情况，判断它是“头部场景记忆”还是“带偏置的共享原型”。
- [ ] 将结论沉淀到任务板，并给出下一步更值得尝试的方案。

## 3. 执行报告 (移入 3_Done 时填写)
- **执行过程**:

  2. 新建 `@examples/kuairand-1k/vq_branch_probe.py`，复用当前 KuaiRand 数据管线，对 `vq_share` 和 `vq_share_gate_v1` 逐 epoch 输出：
     - `full / shared_only / specific_only` AUC
     - 分场景 AUC 差值
     - codebook 激活数、跨场景熵、主场景占比、跨场景覆盖度
  3. 跑通 smoke：
     - `uv run python @examples/kuairand-1k/vq_branch_probe.py --model vq_share_gate_v1 --seed 2024 --max_epochs 1 --device cuda --log_name vq_branch_probe_gate_v1_smoke.log --report_name vq_branch_probe_gate_v1_smoke.json`
  4. 跑正式 2 epoch 对照：
     - `uv run python @examples/kuairand-1k/vq_branch_probe.py --model vq_share_gate_v1 --seed 2024 --max_epochs 2 --device cuda --log_name vq_branch_probe_gate_v1_seed2024.log --report_name vq_branch_probe_gate_v1_seed2024.json`
     - `uv run python @examples/kuairand-1k/vq_branch_probe.py --model vq_share --seed 2024 --max_epochs 2 --device cuda --log_name vq_branch_probe_vq_share_seed2024.log --report_name vq_branch_probe_vq_share_seed2024.json`
  5. 重新核对 `@examples/kuairand-1k/kuairan1k.py` 的数据流，确认当前 `trainer.run()` 实际使用 `valid_loader=test_path(20220508)` 做训练期验证，`20220507` 虽被构建但没有用于 `run()` 内部验证。
- **结果验证**:
  1. 当前训练方式：
     - 当前入口的“验证集”实际上是 `20220508` split。
     - `20220507` 被构建成数据集，但没有进入 `trainer.run()` 的训练期评估路径。
     - 因此本轮 branch probe 的 AUC 结论与现有训练日志是对齐的，都是在 `20220508` split 上比较。
  2. `shared` 和 `specific` 谁在提供收益：
     - `vq_share_gate_v1`，epoch2：
       - `full = 0.786004`
       - `shared_only = 0.736374`
       - `specific_only = 0.672327`
     - `vq_share`，epoch2：
       - `full = 0.786017`
       - `shared_only = 0.734691`
       - `specific_only = 0.669651`
     - 结论：
       - `specific_only` 单独拿出来远弱于 `shared_only`，说明 specific 不是独立可用的主预测器。
       - `shared` 是主干，`specific` 更像 residual correction。
       - `gate_v1` 并没有改变这个基本事实，只是把 correction 用得更激进。
  3. 分场景收益归因，`vq_share_gate_v1` epoch2 相比 `shared_only` 的 full 增益：
       - `domain0(tab0)`: `+0.102513`
       - `domain1(tab1)`: `+0.002460`
       - `domain2(tab2)`: `+0.011690`
       - `domain3(tab4)`: `+0.015494`
       - `domain4`: `-0.014832`
     - 解释：
       - 加上 `specific` 后，最大收益主要来自 `domain0(tab0)`，它是第二大场景，不是真正的最大场景。
       - 最小场景 `domain4` 在 `shared_only` 下反而更好，说明现有 residual 修正对小场景是噪声放大，不是有效补偿。
       - `vq_share` 也呈现同样模式，说明问题不只在门控，而在“specific 分支如何学习场景差异”这件事本身。
  4. VQ 码本里到底存了什么：
     - `vq_share_gate_v1` epoch2：
       - `active_code_count = 64/64`
       - `mean_active_domain_count = 4.953`
       - `codes_used_by_1_domain_ratio = 0.0`
       - `codes_used_by_3plus_domains_ratio = 1.0`
       - `weighted_top_domain_ratio = 0.6835`
       - `weighted_normalized_domain_entropy = 0.5396`
     - `vq_share` epoch2：
       - `active_code_count = 64/64`
       - `mean_active_domain_count = 5.0`
       - `codes_used_by_1_domain_ratio = 0.0`
       - `codes_used_by_3plus_domains_ratio = 1.0`
       - `weighted_top_domain_ratio = 0.6841`
       - `weighted_normalized_domain_entropy = 0.5432`
     - 测试集真实场景先验：
       - `tab1(domain1)` 占比 `0.6828`
       - `tab0(domain0)` 占比 `0.1913`
       - `tab4(domain3)` 占比 `0.0742`
       - `tab2(domain2)` 占比 `0.0358`
       - `tab6(domain4)` 占比 `0.0159`
       - 该先验分布的归一化熵约为 `0.5934`
     - 结论：
       - codebook 不是“每个 code 只服务一个场景”的塌缩记忆，因为没有单场景 code，且所有 code 基本都被 5 个场景共同使用。
       - 但 code 也不是完全场景无关，因为每个 code 的分布熵仍低于整体场景先验，说明单个 code 内部存在一定场景偏置。
       - 更准确的说法是：VQ 存的是“跨场景共享原型”，但这些原型的使用频率仍受头部场景先验支配，是“共享知识 + 频次偏置”，不是纯粹 domain-invariant representation。
  5. Observatory 辅助结论：
     - `vq_share_gate_v1` 的 `gate_series` 显示，训练后期 `gate_specific_mean` 稳定在约 `0.60`，`gate_shared_mean` 约 `0.40`，`gate_entropy` 约 `0.49`，说明门控不是均匀软平均，而是在持续偏向 residual。
     - 同时 `feature_var_shared_ratio` 约 `0.94`，`logit_var_shared_ratio` 约 `0.58 ~ 0.60`，说明表征主体仍然在 `shared`，`specific` 负责较小但足以改变排名的 logit 修正。
     - 需要注意：`observatory/vq_domain_code_usage.png` 来源于 `_latest_debug` 的最新 batch，不应用作全量数据级别结论；全量判断以 branch probe JSON 统计为准。
  6. 下一步更值得做的方案：
     - 优先方向不是继续做更强的 `specific`，而是做“小场景 shared 保护 + residual 再平衡”。
     - 更具体地，可以优先尝试两类轻量方案：
       - `rarity-aware shared floor`：对低频场景抬高 shared 保底，限制 residual 在小场景的放大权。
       - `prior-aware gate regularization`：让 gate 不要在训练后期持续向 `specific` 偏斜，尤其在低频场景上维持更高 shared 权重。
