
## 序列推荐
### 一、留一法LS, 以用户为中心
```python
"eval_args": {
    "split": {"LS": "valid_and_test"}, # 最后两次用来val和test
    "order": "TO",  # 严格按时间 Time Order 排序
    "mode": "full"  # 全排序评测, 对每个用户的所有未记录的历史评分
}
```
1. Hit@10 (对所有用户前10推荐，能够命中label的用户比例)
2. Recall@10 (前 10 名里包含的“真实喜欢的物品数” / “总共真实喜欢的物品数”，但是留一法，目标物品只有一个，所以等于Hit@10)
3. **MRR@10** (平均倒数排名) 和 **NDCG@10** (归一化折损累计收益)：它不仅关心你有没有命中前 10，还关心你**排在第几名**。越靠前，分数越高。

### 二、CTR, 不分组
```python
"eval_args": {
    "split": {"RS": [0.8, 0.1, 0.1]}, # 随机切分
    "group_by": "None",               # 【核心】取消按 user 分组
    "order": "RO",                    # Random Order 随机顺序
    "mode": "labeled"                 # 按照带有 label 的模式进行评测(如果你有显性的正负样本）
                                      # - 隐式的就要让recbole帮你采样
},
# 2. 将评估指标换成 CTR 指标
"metrics": ["AUC", "LogLoss"],
"valid_metric": "AUC",

# 3. 负采样：既然是点对点二分类，框架必须构造负样本（label=0）
"train_neg_sample_args": {"distribution": "uniform", "sample_num": 1},
```
计算AUC，不分组的情况下，recbole会为验证集所有样本调用model.predict得到所有UI对的打分。然后让sklearn计算AUC
. 而sklearn计算AUC必须多传入一个gt参数，表示每个UI对的label是否为1或者0。假如你的原数据只有正样本，那么你必须eval_neg_sample_args来指定负样本的来源。


训练期（Train）： 为了保证正向梯度不被负样本淹没，训练时的负采样比例通常维持在 1:1 到 1:5 之间。有些高阶玩法会加入 Hard Negative Mining（挖掘困难负样本）。

评估期（Eval）： 学术界最主流的做法是 1:99。即对于测试集里的每一个真实的交互（正样本），随机抽取 99 个用户未交互的物品（负样本），组成一个长度为 100 的列表。计算模型在这 100 个样本中的 AUC、Hit@10 和 NDCG@10。这既保证了计算效率，又引入了足够的竞争难度。
